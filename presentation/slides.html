03 — Bối Cảnh

Bối Cảnh Bài Toán

PTITMeet ra đời từ nhu cầu thực tế: một nền tảng họp trực tuyến nội bộ, linh hoạt, đa nền tảng.

🎯 PTITMeet là gì?
Ứng dụng họp trực tuyến được xây dựng từ đầu theo kiến trúc Microservices, hỗ trợ đồng thời:

• Web App — React, chạy trên trình duyệt
• Mobile App — Java/Android
• Backend — Spring Boot Microservices
🔑 Tính năng cốt lõi
• Tạo & tham gia phòng họp tức thì hoặc lên lịch
• Video/Audio/Screen Share qua WebRTC
• Chat real-time trong phòng họp
• Waiting Room — Host duyệt người vào
• Host Controls — Mute, Kick, Transfer Host
• Ghi hình cuộc họp, lưu lên Cloud
• Lịch sử & thống kê sau cuộc họp
🌐 Các bên tham gia hệ thống
👤
Người dùng thông thường (Guest)
Đăng ký tài khoản, tham gia phòng họp qua link/mã, chat, xem lịch sử cuộc họp của mình.
🧑‍💼
Người tạo phòng (Owner)
Tạo và quản lý toàn vòng đời phòng họp, cấu hình quyền truy cập, mời thành viên, xem ghi hình.
🎙️
Host (Người điều phối)
Điều phối phiên họp thực tế: approve waiting room, mute/kick người, chuyển quyền host, kết thúc cuộc họp cho tất cả.
🤖
LiveKit SFU
Thành phần xử lý toàn bộ media (audio/video). Tách biệt khỏi backend — kết nối trực tiếp từ client.
04 — Phát Biểu Bài Toán

Tại Sao Cần Microservices?

Bài toán đặt ra những yêu cầu mà kiến trúc Monolith khó đáp ứng tốt trong dài hạn.

❌ Vấn đề với Monolith
Module quá khác nhau về tính chất
Chat cần MongoDB (document, fast write) · Meeting cần MySQL (relational, transactions) · Recording cần tích hợp LiveKit Egress riêng → Nhồi tất cả vào 1 codebase = phức tạp, khó maintain.
Tải không đều — khó scale
Giờ cao điểm: Meeting Service tải × 10, Identity Service tải bình thường → Monolith buộc phải scale toàn bộ chỉ vì một module.
Deploy rủi ro cao
Sửa 1 dòng logic chat → phải rebuild & redeploy toàn bộ hệ thống → downtime, rủi ro rollback.
✅ Microservices giải quyết
🔧
Loose Coupling — Database per Service
Mỗi service có DB riêng. Thay đổi schema Meeting không ảnh hưởng Chat. Liên kết qua Raw UUID, không có Foreign Key xuyên service.
📈
Scale độc lập từng service
Chỉ scale Meeting Service × 3 mà không cần scale Identity hay Chat.
🛡️
Fault Isolation
Chat Service lỗi không làm Meeting Service sập. Hệ thống degraded gracefully, không down hoàn toàn.
⚡
Deploy độc lập
Sửa Chat Service → chỉ cần build & deploy Chat Service. CI/CD chỉ trigger service nào có thay đổi.

07 — Database per Service

Cơ Sở Dữ Liệu Độc Lập

Không có Foreign Key xuyên service. Liên kết bằng Raw UUID — đặc trưng quan trọng của Loose Coupling.

🗄️ ptitmeet_identity_db (MySQL)
usersuser_id, email, password_hash, full_name, avatar_url, auth_provider
🗄️ ptitmeet_meeting_db (MySQL)
meetingsmeeting_id, host_id, owner_id, meeting_code, status, access_type
participantsuser_id (raw UUID — không FK sang identity_db)
participant_sessionsACTIVE / LEFT / KICKED / ENDED_BY_HOST
meeting_invitationsRESTRICTED access type — danh sách email được mời
outbox_eventsPENDING / SENT — Transactional Outbox cho Kafka
🗄️ ptitmeet_chat_db (MongoDB)
meeting_chatsmeeting_code, sender_id (raw UUID), content, timestamp
🗄️ ptitmeet_media_db (MySQL)
meeting_recordingsegress_id, meeting_id (raw UUID), owner_id, status, file_url
🔑 Raw UUID thay vì Foreign Key
participants.user_id lưu UUID của user nhưng không có FK sang identity_db.users.

→ Khi cần tên user, Meeting Service gọi gRPC sang Identity Service.
→ Hai DB có thể chạy trên server khác nhau — không phụ thuộc lẫn nhau.
⚡ Redis — Identity Service Token Store
⚡ Redis In-Memory Store
auth:refresh:{id}Refresh Token · TTL 30 ngày · Token Rotation
auth:reset:{token}Password Reset · TTL 15 phút · Single-use
auth:blacklist:{jti}JWT Blacklist (logout) · TTL = remaining JWT lifetime
📌 Access Type của phòng họp
OPEN — Ai có link đều vào được
TRUSTED — Chỉ email đúng domain (vd: @ptit.edu.vn)
RESTRICTED — Chỉ email trong danh sách được mời

Dùng khi cần kết quả ngay lập tức để xử lý tiếp. Ưu tiên gRPC cho inter-service call nội bộ.

⚖️ Compensating Transaction
Meeting gọi Media → start recording thành công.
Sau đó Meeting DB fail → Rollback local DB
→ Compensate: DELETE /recordings/{egressId}
→ Media: stop egress + xóa bản ghi. Hệ thống consistent.
🌐 Luồng API Gateway → Service
1
Client gửi Authorization: Bearer <token>
Request HTTPS đến Gateway :8080
2
Gateway verify JWT (Shared Secret)
Không cần gọi Identity Service — verify cục bộ bằng HS256
3
Inject headers vào request
X-User-Id & X-User-Email — service con đọc header, không verify JWT lại
4
Forward tới service đích
Dựa theo URL prefix: /api/meetings → Meeting Service :8082
🔍 Service Discovery (Eureka)
Gateway dùng lb://meeting-service thay hard-coded URL.
Eureka cập nhật registry khi service up/down.
gRPC client resolve địa chỉ server dynamically.
09 — Asynchronous Communication

Tại Sao Cần Bất Đồng Bộ (Kafka)?

Không phải luồng nào cũng cần đợi kết quả. Kafka giải phóng người dùng khỏi những tác vụ nền tốn thời gian.

❓ Bài toán: Lên lịch họp & gửi email mời
❌ Nếu gửi email ngay trong request (đồng bộ)
User nhấn "Tạo lịch họp" → Server phải đợi gửi email cho 50 người → Request treo 10–30 giây → Timeout → Trải nghiệm tệ.

Tệ hơn: SMTP server tạm lỗi → Cả việc tạo lịch họp cũng thất bại, dù dữ liệu đã được lưu.
✅ Với Apache Kafka (bất đồng bộ)
User nhấn "Tạo lịch họp" → Server lưu DB → Trả về thành công ngay.

Kafka chạy nền: đọc event → Notification Service gửi email lần lượt → Tách biệt hoàn toàn khỏi luồng chính. SMTP lỗi không ảnh hưởng việc tạo lịch.
📋 Kafka dùng ở đâu trong PTITMeet?
• Lên lịch họp → gửi email mời participants
• Hủy lịch họp → gửi email thông báo
• Recording hoàn thành → gửi link download
🔒 Outbox Pattern — Đảm bảo Event Không Mất
Vấn đề: Ghi DB thành công → publish Kafka thất bại (Kafka tạm down) → email không được gửi, không có cách retry.
Giải pháp Outbox Pattern: Ghi event vào DB cùng transaction với dữ liệu nghiệp vụ. Nếu DB commit OK → event chắc chắn tồn tại.
1
Một transaction duy nhất
INSERT meetings + INSERT outbox_events(PENDING) → COMMIT → không thể tách rời
2
Background Worker liên tục poll
Quét outbox_events PENDING → Publish Kafka → Đánh dấu SENT
3
Kafka down? Không sao!
Event vẫn ở outbox_events PENDING → Worker sẽ retry lần sau khi Kafka hoạt động lại
4
Notification Service consume
Đọc event từ Kafka → Gửi email → Guaranteed at-least-once delivery
10 — Real-time Communication

WebSocket, STOMP & LiveKit

Hai kênh real-time phục vụ hai loại dữ liệu hoàn toàn khác nhau — signaling và media.

💬
WebSocket + STOMP
Client ↔ Backend (Chat Service)
WebSocket là giao thức kết nối hai chiều liên tục giữa client và server — khác HTTP ở chỗ không cần gửi request mới mỗi lần, server có thể chủ động push dữ liệu bất cứ lúc nào.

STOMP là tầng messaging chạy trên WebSocket, cung cấp cơ chế pub/sub theo topic — client subscribe vào topic, server broadcast, ai đang subscribe đều nhận được.
Các loại sự kiện qua STOMP:
Chat message: client gửi → server lưu MongoDB → broadcast cho cả phòng
Waiting room: Guest xin vào → Host nhận notification → Approve/Reject → Guest nhận token
Host controls: MUTE / KICK / TRANSFER_HOST → broadcast cho toàn phòng
Recording events: RECORDING_STARTED / STOPPED
🎙️
LiveKit WebRTC (SFU)
Client ↔ LiveKit Server (không qua backend)
WebRTC là công nghệ truyền media (audio/video) peer-to-peer, độ trễ cực thấp, không qua backend server.

SFU (Selective Forwarding Unit): LiveKit đóng vai trò trung gian — mỗi client chỉ gửi 1 stream lên SFU, SFU phân phối tới các client còn lại. Không cần mesh P2P khi có nhiều người.
Sự kiện đi thẳng Client → LiveKit (không qua backend):
Giơ tay / thả cảm xúc — LiveKit Data Channel, broadcast tới toàn phòng qua SFU
Bật/tắt camera, mic — LiveKit track publish/unpublish, latency thấp
Chia sẻ màn hình — screen capture stream gửi thẳng lên SFU
Audio/Video call — media stream, không qua WebSocket backend
🔑 Tại sao tách 2 kênh?
STOMP qua backend: phù hợp signaling cần lưu lịch sử (chat, approval).
LiveKit WebRTC: phù hợp media cần độ trễ thấp <100ms — không thể đi qua backend.
11 — Third-party Integration

Tích Hợp Bên Thứ Ba

4 tích hợp bên ngoài — mỗi cái giải quyết một bài toán chuyên biệt mà không cần tự xây dựng.

🎙️
LiveKit SFU + Egress
Hạ tầng WebRTC. Meeting Service phát Join Token → client dùng token kết nối SFU trực tiếp.

Media Service gọi Egress API để ghi hình phòng họp → LiveKit encode & upload → AWS S3.

LiveKit gửi Webhook khi recording hoàn thành → Media Service cập nhật file_url.
🔵
Google OAuth 2.0
Identity Service nhận idToken từ Google SDK (Web & Android).

Verify với Google Public Key → Lấy email, name → Tạo hoặc link tài khoản → Trả về Access Token + Refresh Token.

Người dùng không cần nhớ password — đăng nhập bằng Google account.
☁️
AWS S3
Lưu trữ file video recording. LiveKit Egress tự động upload lên S3 sau khi ghi xong — Media Service chỉ nhận URL qua webhook.

Không cần server storage riêng. File_url trả về client để download trực tiếp từ S3.
📧
Email SMTP (Notification)
Notification Service consume Kafka events.

Gửi email mời họp khi lên lịch, email hủy lịch, email reset password.

Hoàn toàn tách biệt khỏi luồng chính — người dùng không phải đợi email được gửi mới nhận response.

13 — Tổng Kết

Đã Làm Được & Bài Học Rút Ra

Những đánh đổi thực tế khi xây dựng hệ thống Microservices — không có quyết định nào là hoàn hảo tuyệt đối.

✅ Đã hoàn thành
✅5 Microservices độc lập, Database per Service
✅Kafka + Transactional Outbox Pattern
✅gRPC inter-service communication
✅WebSocket/STOMP real-time + LiveKit WebRTC
✅Cross-platform: Web React + Android Java
✅CI/CD Pipeline chỉ build service có thay đổi
✅Docker Compose containerize toàn bộ stack
💡 Bài học & Đánh đổi thực tế
⚖️ Monorepo vs Polyrepo: Common library dùng chung rất tiện, nhưng đánh đổi là chỉ cần sửa library → toàn bộ service phải build lại. Cần cân nhắc kỹ trước khi chia sẻ code giữa services.
🔄 Bất đồng bộ không phải lúc nào cũng tốt: Kafka tăng độ phức tạp đáng kể. Chỉ nên dùng async khi thực sự không cần kết quả ngay & chấp nhận eventual consistency.
🏗️ Phân chia service hợp lý: Không nên tạo service riêng cho từng entity (Anti-pattern). Phân chia theo bounded context nghiệp vụ — Meeting, Identity, Chat là những domain thực sự độc lập.
🗄️ Chọn đúng database: Chat dùng MongoDB (document, write-heavy, không cần join) · Meeting/Identity dùng MySQL (relational, ACID transaction). Mỗi bài toán có loại DB phù hợp.
🚀 CI/CD thông minh: Detect service nào thay đổi qua git diff → Chỉ build & deploy service đó. Giảm thời gian pipeline từ ~10 phút xuống ~2 phút cho thay đổi nhỏ.
